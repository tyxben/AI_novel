# AI 长篇小说写作模块 - 叙事节奏控制 v2 需求文档

## 1. 目标

为 AI 长篇小说生成系统增加三层叙事控制机制，解决当前系统在 **卷级进度把控**、**金手指失效机制**、**策略复杂度升级** 三个维度的缺失，确保生成的长篇小说在 100-200 章规模下能够保持结构张力、机制多样性和内容复杂度的持续升级，避免中段失速、机制僵化和内容重复的问题。

---

## 2. 背景：三大结构性缺陷的实证观察

当前小说生成系统基于静态大纲 + 章节级连续性检查，在短篇（<30 章）表现良好，但在长篇（100+ 章）创作中暴露三个系统性失效模式。以下观察均来自实际生成项目 `workspace/novels/novel_12e1c974`（玄幻战争题材，目标 200 章 / 50 万字，已生成 27 章）：

### 2.1 问题 1：卷切换不强制执行，中段失速

**现象**：
- **大纲承诺**：第 1 卷"边村立旗"（ch1-25）= 从草莽到领主，第 2 卷"群山开疆"（ch26-60）= 多山域扩张战争
- **实际生成**：
  - ch10 已经完成第 1 卷高潮（攻打青云门、接管山门矿脉），比计划提前 15 章
  - ch11-27 陷入单一场景死循环：林辰困在矿场处理内鬼、应对黑风寨骚扰、盘查矿工、审讯卧底，场景尺度从未扩大
  - ch27 仍在"矿场外缘三十多米处""废道口""木棚"等同一套地点反复推演
  - 第 2 卷主题"群山开疆"在 ch26 根本未启动，ch27 的叙事规模仍停留在第 1 卷中段水平

**症状根因**：
1. `PlotPlanner` 每章规划时只看 `continuity_brief`（上章遗留钩子 + 未兑现债务 + 角色状态），无法感知"当前卷已消耗 X% 章节但 Y% 关键里程碑未完成"
2. `volume_settlement.py` 仅在卷末倒数 3 章才触发 `settlement_prompt`，且只检查"债务履行"而不检查"卷主题里程碑是否完成"
3. 大纲的 `volume.theme` / `volume.climax` / `volume.end_hook` 被视为"建议"而非"预算"，无硬性执行机制

**数据证据**：
- 第 1 卷大纲：`climax: "林辰借赵青云失踪留下的破绽，反杀青云门残余势力，并在第24-25章中守住青云山门"`
- 实际：ch10 标题"青云门覆灭"，ch25 距离 ch10 已过 15 章，但仍在处理矿场内务，青云山门早在 ch10 就"正式接管"
- 第 2 卷大纲：`theme: "扩张与治理并行，分封初成，宗门压力升级"`
- 实际：ch26-27 叙事尺度仍为"一个矿场 + 周边村镇"，无任何"群山""分封""宗门压力升级"迹象

### 2.2 问题 2：系统（金手指）无失效机制，零张力

**现象**：
- **用户主题显式要求**：`novel.json` 中 `theme: "系统经常出现故障或并没有用那么厉害只是保底"`
- **实际生成 27 章系统表现**：
  - ch3: 系统提供"地图扫描""资源识别"，100% 精准定位敌人位置和矿脉
  - ch5: 战术模拟功能给出"82% 成功率"的声东击西方案，林辰完全依赖此数据制定战术
  - ch8: 系统提供"灵力波动分析"，揭示赵青云消失位置
  - ch10-27: 系统持续提供"矿脉透视""倒计时""破解禁制""灵力强度分析"，从未误判、从未失效、从未给出错误信息
- **后果**：
  - 主角决策完全依赖系统数据，无需判断、无需冒险、无需 improvisation
  - 读者零悬念："系统说能成"="必然成功"
  - 主题承诺的"帝王心术""兵法智慧"无展现空间，因为系统已提供最优解

**症状根因**：
1. Writer 的系统提示中，系统被描述为"能力"而非"有状态的工具"，无"可用性状态""误差范围""冷却时间"等字段
2. PlotPlanner 无权调度"本章系统失效"事件，`chapter_brief` 中无"系统状态"字段
3. NovelDirector 在大纲规划时未预留"系统失效窗口"（如"第 8 章系统给出错误敌军数量导致战术失败"）
4. 无数据结构追踪系统的历史表现（成功次数、失败次数、上次失效章节）

**数据证据**：
- 27 章中系统提及次数：34 次，准确率：34/34 = 100%
- 从未出现："系统扫描失败""数据错误""功能暂时不可用""系统提示模糊不清"
- 主角从未因系统失误而陷入危机

### 2.3 问题 3：策略复杂度停滞，内容重复

**现象**：
- **用户主题显式要求**：`theme: "需要战略战术，参考各种兵法，持久战，心理战，36计等等兵法。需要分封，需要帝王心术。"`
- **实际生成的策略手段**（ch1-27）：
  - ch5: "声东击西"（36 计第 6 计）
  - ch10: "声东击西 + 藏经阁次级节点"（重复 ch5 策略 + 小变种）
  - ch14: "立规矩，杀鸡儆猴"
  - ch20-27: 反复使用"抓内奸""审讯""布置伏点""卡哨位""诱敌""钓鱼执法"，均为战术级小聪明
  - **从未出现**：
    - 36 计的其他 35 计（美人计、反间计、围魏救赵、釜底抽薪、欲擒故纵等）
    - 兵法术语（孙子兵法、持久战、心理战）
    - 帝王心术行为（分封、联姻、削藩、制衡、养寇自重、恩威并施）
    - 战略级行为（粮道封锁、情报战、经济战、联盟外交）
- **后果**：
  - ch5 和 ch20 的策略层次感相同，无升级
  - 读者审美疲劳："又是抓内奸""又是矿场伏击"
  - 主题承诺的"兵法大师成长弧线"不存在

**症状根因**：
1. PlotPlanner 无"策略复杂度分级"概念，选择策略时无"本章必须使用 Tier X 策略"约束
2. Writer 的训练数据偏向小规模战术（容易生成且不易出错），自然退化到舒适区
3. 无"策略元素库"供 PlotPlanner 强制抽取（如"本章必须使用 36 计中的 {未使用计谋}"）
4. 无"重复惩罚"机制（同一策略元素在 N 章内禁止重复使用）
5. 无"阶梯映射"（如"第 1-30 章限定 Tier 1-2 策略，第 31-60 章必须进入 Tier 3-4"）

**数据证据**：
- 27 章中提及"声东击西"：4 次
- 提及 36 计其他计谋：0 次
- 提及"孙子兵法""持久战""心理战"：0 次
- 提及"分封""削藩""制衡"等帝王术：分封在 ch6 提及 1 次后再未出现

## 3. 核心需求清单

### 3.1 需求组 A：卷进度预算与里程碑强制执行

#### R-A-1: 卷级里程碑定义（P0）

**功能描述**：
- 在大纲生成阶段（`NovelDirector.generate_outline()`），为每个 `VolumeOutline` 分配 3-5 个 **可验证的里程碑事件**（`narrative_milestones`）
- 每个里程碑包含：
  - `milestone_id`: 唯一标识
  - `description`: 中文描述（如"主角攻占第一个宗门据点""分封三名将领建立封地制度""系统第一次重大失效导致战术失败"）
  - `target_chapter_range`: 目标完成章节范围（如 [15, 20] 表示应在第 15-20 章之间完成）
  - `verification_type`: 验证方式，`auto_keyword`（关键词匹配）或 `llm_review`（LLM 判定）
  - `verification_criteria`: 验证条件（关键词列表 或 LLM 提示词）
  - `priority`: 优先级 `critical`（卷末必完成）/ `high`（建议完成）/ `normal`（可延后）
- 里程碑由 LLM 根据 `volume.theme` / `volume.climax` / `volume.end_hook` 自动生成，可人工调整

**WHY**：
- 大纲的 `theme` / `climax` 是自然语言，无法被代码直接检查。里程碑将其转化为可量化的检查点
- 里程碑范围允许灵活性（如"15-20 章"），但避免无限拖延

**验收标准**：
- 为一个 200 章小说的 6 个卷生成 18-30 个里程碑
- 每个里程碑的 `target_chapter_range` 不重叠度 > 70%（允许部分里程碑跨章节并行）
- 80% 的里程碑可用关键词验证（如"占领青云山门"→ 检查章节是否包含"青云山门""攻占""接管"），20% 需要 LLM 判定
- 里程碑存储在 `novel.json` 的 `volumes[].narrative_milestones` 字段

#### R-A-2: 实时进度追踪（P0）

**功能描述**：
- 每章生成后，自动检查该章是否完成了某个待完成的里程碑
- 检查逻辑：
  - 对于 `auto_keyword` 类型，扫描 `chapter.full_text` 是否包含所有关键词
  - 对于 `llm_review` 类型，调用 LLM 判定（传入里程碑描述 + 章节摘要，返回 `true/false`）
- 完成的里程碑标记为 `status: "completed"`，记录 `completed_at_chapter`
- 未完成的里程碑若超过 `target_chapter_range.max`，标记为 `status: "overdue"`

**WHY**：
- 无自动检查，里程碑形同虚设
- LLM 检查仅用于复杂语义判断，大部分里程碑用关键词即可快速验证（省 token）

**验收标准**：
- 自动检查延迟 < 10 秒/章（关键词检查）或 < 30 秒/章（LLM 检查）
- 对人工植入的已完成里程碑，检测准确率 >= 90%
- 对未完成里程碑，误报率 < 10%（不能把未完成的标记为完成）

#### R-A-3: 进度注入到 `continuity_brief`（P0）

**功能描述**：
- `ContinuityService.generate_brief()` 新增字段：
  - `volume_progress`: dict，包含：
    - `current_volume`: 当前卷编号 + 标题
    - `chapters_consumed`: 本卷已使用章节数
    - `chapters_remaining`: 本卷剩余章节数
    - `milestones_completed`: 已完成里程碑列表（简述）
    - `milestones_pending`: 待完成里程碑列表，按优先级 + 逾期状态排序
    - `milestones_overdue`: 逾期里程碑列表（高亮）
    - `progress_health`: 进度健康度，`on_track`（已完成 >= 50% 里程碑且无逾期）/ `behind_schedule`（已完成 < 50% 且无逾期）/ `critical`（有逾期里程碑）
- `ContinuityService.format_for_prompt()` 将 `volume_progress` 格式化为中文提示块：
  ```
  ### 当前卷进度摘要
  - 卷：第 1 卷"边村立旗"（已用 18/25 章，剩余 7 章）
  - 已完成里程碑：✓ 激活系统，✓ 招募第一批追随者
  - 待完成里程碑：攻占青云山门（目标 ch15-20，当前 ch18，接近超期）
  - **逾期里程碑：建立分封制度（目标 ch20-25，当前 ch18，已逾期）**
  - 进度状态：behind_schedule — 本卷 40% 里程碑已完成，需加速推进剩余关键事件
  ```
- 此提示块注入 `PlotPlanner` 和 `Writer` 的系统提示

**WHY**：
- `PlotPlanner` 需要知道"我规划的本章场景是否推进了卷级目标"
- `Writer` 需要知道"剩余章节不多，必须在本章完成 X 里程碑"
- 明确的进度压力避免中段失速

**验收标准**：
- `continuity_brief` 包含完整 `volume_progress` 字段
- 提示块长度 < 500 字（避免过度占用 context）
- PlotPlanner 收到提示后，在逾期里程碑存在时，生成的场景必须明确推进该里程碑（人工审查 10 章样本，符合率 >= 80%）

#### R-A-4: PlotPlanner 响应进度信号（P0）

**功能描述**：
- `PlotPlanner.decompose_chapter()` 在场景分解前，读取 `volume_progress`
- 决策逻辑：
  - 若 `progress_health == "on_track"`：正常分解，允许引入新支线
  - 若 `progress_health == "behind_schedule"`：优先推进 `milestones_pending` 中的第一个里程碑，抑制新支线
  - 若 `progress_health == "critical"`（有逾期）：本章至少 1 个场景必须直接服务于逾期里程碑，强制推进
- 在 `_DECOMPOSE_USER` prompt 中增加节：
  ```
  ## 卷进度约束
  {volume_progress_constraint}
  ```
  其中 `volume_progress_constraint` 根据进度状态生成：
  - `on_track`: "当前卷进度正常，本章可正常推进主线或适度展开支线"
  - `behind_schedule`: "当前卷进度落后，本章必须优先推进以下里程碑：{pending_milestones_top3}，避免引入与里程碑无关的新支线"
  - `critical`: "**警告**：以下里程碑已逾期：{overdue_milestones}，本章必须有至少 1 个场景直接推进其中一个逾期里程碑，不允许与里程碑无关的场景"

**WHY**：
- 不强制 PlotPlanner 响应，进度信号无效
- 明确的约束 prompt 比隐式期待更有效

**验收标准**：
- 当有逾期里程碑时，PlotPlanner 生成的场景中至少 1 个场景的 `summary` 明确提及逾期里程碑相关事件（人工审查 10 个案例，符合率 >= 90%）
- 当 `on_track` 时，PlotPlanner 可自由分配场景内容（不影响创作灵活性）

#### R-A-5: 卷边界强制收束（P1）

**功能描述**：
- 增强现有 `VolumeSettlement.get_settlement_brief()`：
  - 除了债务收束外，增加 **里程碑完成度检查**
  - 若卷末倒数 3 章内仍有 `priority: critical` 的未完成里程碑，在 `settlement_prompt` 中高亮警告
  - 若卷末仍有未完成的 `critical` 里程碑，`VolumeSettlement` 可选择：
    1. **继承到下卷**：将里程碑移入下一卷的 `milestones_pending`，标记为 `inherited: true`
    2. **标记为失败**：记录决策日志，允许故事继续（某些未完成里程碑可能因情节变化而放弃）
- 在卷末最后一章生成后，运行 `VolumeSettlement.settle_volume()`：
  - 生成卷完成度报告（已完成 / 未完成 / 继承 / 放弃里程碑清单）
  - 写入 `novel.json` 的 `volumes[].settlement_report`
  - 若下一卷存在，将继承里程碑注入下一卷的 `narrative_milestones`

**WHY**：
- 卷边界是强制校准点，避免累积偏差
- 明确继承或放弃机制，避免未完成里程碑悬而未决

**验收标准**：
- 卷末生成的 `settlement_report` 包含里程碑完成度统计（百分比）
- 若卷末有 2 个 `critical` 里程碑未完成，报告中明确标注"继承到第 X 卷"或"故事走向变化，里程碑 Y 已放弃"
- 继承的里程碑在下一卷的 `milestones_pending` 中可见

### 3.2 需求组 B：系统能力状态机（金手指失效机制）

#### R-B-1: 系统状态数据模型（P0）

**功能描述**：
- 新增 Pydantic 模型 `SystemCapabilityState`（存储在 `novel.json` 的 `system_state` 字段）：
  ```python
  class SystemCapabilityState(BaseModel):
      current_mode: Literal["full", "degraded", "offline", "wrong_data", "limited"]
      degradation_reason: str | None  # "灵力干扰" / "系统过载" / "未知故障"
      affected_capabilities: list[str]  # ["地图扫描", "战术模拟", "灵力分析"]
      recovery_trigger: str | None  # "到达安全区" / "3 章后自动恢复" / "完成系统修复任务"
      recovery_chapter: int | None  # 预期恢复章节号
      failure_history: list[dict]  # [{chapter, mode, reason}, ...]
  ```
- 模式定义：
  - `full`: 系统全功能（默认）
  - `degraded`: 部分功能降级（如"扫描范围缩小到 50%""数据刷新延迟 10 分钟"）
  - `offline`: 系统完全失联
  - `wrong_data`: 系统给出错误数据（如"敌军 50 人"实际 200 人）
  - `limited`: 功能可用但有冷却时间（如"本章已使用 1 次扫描，剩余次数 0"）

**WHY**：
- 将系统从"永久能力"转为"有状态的工具"
- `failure_history` 用于后续统计（如"系统在前 50 章失效 5 次，成功 45 次，可信度 90%"）

**验收标准**：
- `SystemCapabilityState` 可序列化到 `novel.json`
- 字段验证规则生效（如 `current_mode` 只能是 5 个枚举值）

#### R-B-2: 大纲阶段预排失效窗口（P0）

**功能描述**：
- `NovelDirector.generate_outline()` 在生成章节大纲后，调用 `schedule_system_failures()`：
  - 输入：`total_chapters`, `volumes`, `climax_chapters`（每卷高潮章节号）
  - 输出：`system_failure_schedule` 列表，每项包含：
    - `chapter`: 失效发生章节
    - `mode`: 失效模式
    - `reason`: 失效原因（中文，如"进入上古禁地，灵力紊乱"）
    - `duration_chapters`: 持续章节数（1-5）
    - `recovery_trigger`: 恢复条件
  - 调度策略（可配置）：
    - **密度**：平均每 15-25 章发生 1 次失效（总失效次数 = `total_chapters / 20`）
    - **时机**：优先安排在卷高潮前 2-5 章（制造"危机前系统失灵→主角必须靠智慧/运气过关"张力）
    - **模式分布**：60% `degraded`，20% `wrong_data`，15% `offline`，5% `limited`
    - **避让规则**：
      - 连续 3 章内不重复失效（避免"系统完全不可用"）
      - 第 1-5 章不失效（新手保护期）
      - 卷末收束章（倒数 2 章）不失效（避免收束混乱）
- 排程结果存储在 `novel.json` 的 `system_failure_schedule` 字段

**WHY**：
- 预先排程确保失效有计划，不是随机崩溃
- 高潮前失效创造最大张力（主角必须 improvise）
- 避让规则防止破坏叙事流畅性

**验收标准**：
- 为 200 章小说生成 8-13 个失效窗口
- 失效窗口中 70% 分布在各卷高潮前 2-5 章
- 无连续 3 章都失效的情况
- 第 1-5 章和每卷倒数 2 章无失效

#### R-B-3: PlotPlanner 读取失效排程（P0）

**功能描述**：
- `PlotPlanner.decompose_chapter()` 在场景分解前，检查 `system_failure_schedule`：
  - 若当前章节在排程中，提取该失效事件的 `mode`, `reason`, `affected_capabilities`
  - 将失效信息注入场景分解 prompt：
    ```
    ## 系统状态约束
    本章系统状态：{mode}
    原因：{reason}
    受影响能力：{affected_capabilities}
    
    **重要**：本章主角无法使用以上系统功能，场景必须体现主角在无系统辅助情况下如何应对挑战。
    ```
  - PlotPlanner 必须生成至少 1 个场景体现"系统失效导致的困境"
  - 若失效模式是 `wrong_data`，必须有 1 个场景体现"主角基于错误数据做决策→遭遇意外"

**WHY**：
- 排程存在但 PlotPlanner 不响应 = 无效
- 明确约束 prompt 强制生成"系统失效"场景

**验收标准**：
- 当排程中有失效事件时，该章生成的场景中至少 1 个场景的 `summary` 明确提及系统失效或功能受限（人工审查 10 个案例，符合率 >= 90%）
- `wrong_data` 模式下，场景必须包含"数据误导→后果"因果链

#### R-B-4: Writer 禁止使用失效功能（P0）

**功能描述**：
- `Writer.write_scene()` 的系统提示中，动态注入当前章节的系统状态：
  - 若系统正常（`mode: full`），正常描述系统能力
  - 若系统失效，明确列出 **禁止使用的功能**：
    ```
    **本章系统状态异常**：
    - 状态：{mode}（{reason}）
    - 禁用功能：{affected_capabilities}
    - 恢复条件：{recovery_trigger}
    
    **写作约束**：
    - 不允许主角使用以上禁用功能
    - 不允许出现"林辰打开系统界面""系统扫描显示"等描述（针对禁用功能）
    - 主角必须依靠自身判断、经验、或其他角色辅助来应对挑战
    ```
- ConsistencyChecker 增加检查规则：若本章系统失效，但章节文本中出现禁用功能的使用（关键词匹配），标记为违规

**WHY**：
- Writer 的自回归生成容易"遗忘"约束，显式禁止列表 + 后验检查双保险
- ConsistencyChecker 的检查防止 Writer 偷偷用系统

**验收标准**：
- 系统失效章节中，Writer 生成的文本不包含禁用功能关键词（如失效"地图扫描"时，文本不包含"系统扫描显示敌军位置"）
- ConsistencyChecker 对人工植入的违规使用（如失效期间写"系统扫描成功"），检测准确率 >= 90%

#### R-B-5: 自动状态转移与恢复（P1）

**功能描述**：
- 每章生成后，`ObligationTracker`（或新模块 `SystemStateTracker`）检查：
  - 若当前章节 = `recovery_chapter`，自动将 `system_state.current_mode` 恢复为 `full`
  - 若 `recovery_trigger` 为条件型（如"到达安全区"），检查章节摘要是否满足条件（LLM 判定或关键词）
  - 恢复时记录到 `failure_history`
- 支持手动触发恢复（用户通过 API 或 CLI 强制恢复，应对剧情变化）

**WHY**：
- 自动恢复避免人工干预
- 条件触发增加灵活性（主角完成特定任务后系统恢复，叙事连贯）

**验收标准**：
- 排程中设定"3 章后恢复"的失效，在第 3 章后 `system_state.current_mode` 自动变为 `full`
- 条件恢复（如"到达安全区"）在章节摘要包含"安全区"关键词后自动恢复

### 3.3 需求组 C：策略复杂度阶梯

#### R-C-1: 策略分级体系定义（P0）

**功能描述**：
- 定义 7 级策略复杂度阶梯（Tier 1-7），存储在 `src/novel/templates/strategy_tiers.yaml`：
  - **Tier 1 — 个体战术**：肉搏、小聪明、单人行动
  - **Tier 2 — 小队战术**：声东击西、伏击、小队协同、简单陷阱
  - **Tier 3 — 战役战术**：情报战、粮道封锁、内部分化、多兵种配合、地形利用
  - **Tier 4 — 制度建设**：分封制、监察体系、税收制度、军功爵位、法律体系
  - **Tier 5 — 战略战役**：大规模战役、持久战、心理战、舆论战、经济封锁
  - **Tier 6 — 高级谋略**：36 计级全局谋略、联盟外交、远交近攻、合纵连横、反间计
  - **Tier 7 — 帝王之术**：制度设计、文化整合、异界整合、宗教统战、历史叙事权
- 每个 Tier 包含：
  - `tier_number`: 1-7
  - `tier_name`: 中文名
  - `description`: 描述
  - `example_elements`: 示例策略元素列表（30-50 个具体策略/计谋/术语）
  - `applicable_scales`: 适用叙事尺度（如 Tier 1: "单人/小场景"，Tier 6: "国家/跨区域"）

**WHY**：
- 明确分级使"复杂度升级"可量化
- 元素库供后续随机/强制抽取

**验收标准**：
- 7 个 Tier 定义完整，每个 Tier 至少 20 个 `example_elements`
- Tier 1-7 的 `example_elements` 无重复（或重复率 < 10%）
- 文件可被 Python 加载为 dict

#### R-C-2: 策略元素注册表（P0）

**功能描述**：
- 新增 `src/novel/templates/strategy_registry.yaml`，存储所有可用策略元素：
  - **36 计清单**：每计包含名称、简述、适用场景、示例
  - **兵法术语**：孙子兵法 13 篇关键概念、持久战、心理战、情报战、粮道战等
  - **帝王心术行为**：分封、联姻、削藩、制衡、养寇自重、恩威并施、挟天子以令诸侯、推恩令、科举制、文字狱等
  - **战略级行为**：联盟外交、经济封锁、文化同化、宗教统战、历史叙事权、移民政策等
- 每个元素包含：
  - `element_id`: 唯一标识
  - `name`: 中文名
  - `tier`: 所属 Tier
  - `description`: 简述
  - `usage_constraints`: 使用约束（如"需要至少 2 个势力存在""需要主角控制区域 >= 3 个城市"）
  - `keywords`: 关键词列表（用于后验检查是否使用）
- 支持 LLM 自动生成（输入题材 + Tier，输出元素列表）+ 人工审核

**WHY**：
- 注册表是强制抽取和重复检测的基础
- `usage_constraints` 防止不合理策略（如"分封制"用在只有 1 个村子的时候）

**验收标准**：
- 注册表包含至少 150 个策略元素
- 36 计全部录入
- 每个元素的 `keywords` 至少 2 个
- 元素按 Tier 分布：Tier 1-2 占 30%，Tier 3-5 占 50%，Tier 6-7 占 20%

#### R-C-3: 卷级 Tier 范围映射（P0）

**功能描述**：
- `NovelDirector.generate_outline()` 在生成卷大纲时，为每个卷分配 `strategy_tier_range`：
  - 输入：卷编号、卷主题、总卷数
  - 输出：`[min_tier, max_tier]`，如第 1 卷 [1, 3]，第 2 卷 [2, 4]，第 6 卷 [5, 7]
  - 映射规则（可配置）：
    - 前 1/3 卷：Tier 1-3
    - 中 1/3 卷：Tier 3-5
    - 后 1/3 卷：Tier 5-7
    - 允许 ±1 Tier 重叠（避免硬切换）
- 存储在 `volumes[].strategy_tier_range`

**WHY**：
- 卷级映射确保整体升级趋势
- 重叠允许回到低 Tier 策略（如高潮后的日常章节用 Tier 2 放松节奏）

**验收标准**：
- 为 6 卷小说分配的 Tier 范围呈递增趋势
- 相邻卷的 Tier 范围有 1-2 级重叠
- 第 1 卷的 `min_tier` = 1，最后一卷的 `max_tier` = 7

#### R-C-4: 章节级策略元素分配（P0）

**功能描述**：
- `PlotPlanner.decompose_chapter()` 在场景分解前，从策略注册表中抽取本章必须使用的策略元素：
  - 输入：当前章节号、当前卷的 `strategy_tier_range`、已使用元素历史（近 5 章）
  - 抽取逻辑：
    1. 在 `strategy_tier_range` 内随机选择一个 Tier（概率分布：中位 Tier 概率最高）
    2. 从该 Tier 的元素库中排除近 5 章已使用的元素
    3. 检查剩余元素的 `usage_constraints`，过滤不满足条件的元素
    4. 随机选择 1-2 个元素作为本章的 `required_strategy_elements`
  - 抽取结果注入到 `chapter_brief`：
    ```python
    chapter_brief = {
        ...
        "required_strategy_tier": 4,
        "required_strategy_elements": [
            {"element_id": "36ji_07", "name": "无中生有", "description": "...", "keywords": ["无中生有", "虚构", "假情报"]},
            {"element_id": "emperor_03", "name": "分封制", "description": "...", "keywords": ["分封", "封地", "藩王"]}
        ]
    }
    ```
- PlotPlanner 在 `_DECOMPOSE_USER` prompt 中增加节：
  ```
  ## 策略要求
  本章必须体现以下策略手段（Tier {tier}）：
  {for element in required_strategy_elements}
  - **{element.name}**：{element.description}
  {endfor}
  
  场景设计时，至少 1 个场景必须明确使用以上策略，并在 summary 中体现。
  ```

**WHY**：
- 强制抽取避免 Writer 总是选择舒适区策略
- 重复检测（排除近 5 章）避免审美疲劳
- `usage_constraints` 避免不合理策略

**验收标准**：
- PlotPlanner 生成的场景中，至少 1 个场景的 `summary` 包含 `required_strategy_elements` 中的关键词（人工审查 10 个案例，符合率 >= 85%）
- 近 5 章内不出现重复的策略元素（检查 20 章样本，重复率 < 10%）
- 策略元素的 Tier 与卷的 `strategy_tier_range` 匹配

#### R-C-5: Writer 强制使用策略元素（P0）

**功能描述**：
- `Writer.write_scene()` 的系统提示中，注入本章的 `required_strategy_elements`：
  ```
  **本章策略要求**：
  本章必须体现以下策略手段：
  {for element in required_strategy_elements}
  - **{element.name}**：{element.description}
    - 关键要素：{element.keywords}
  {endfor}
  
  **写作要求**：
  - 至少在 1 个场景中明确使用以上策略
  - 不要生硬说教（如"林辰想起了三十六计中的 X 计"），而是自然融入情节
  - 策略的使用必须有因果：为什么用这个策略？结果如何？
  ```
- QualityReviewer 增加检查项：若 `required_strategy_elements` 存在，但章节文本中无任何关键词，扣分并标记"策略要求未满足"

**WHY**：
- Writer 单靠 PlotPlanner 的场景分解可能遗忘策略要求，直接注入 Writer prompt 双保险
- QualityReviewer 的检查防止策略流于表面

**验收标准**：
- 章节文本中至少包含 `required_strategy_elements` 的 1 个关键词（关键词检查，覆盖 20 章样本，命中率 >= 90%）
- QualityReviewer 对人工植入的"缺失策略"章节（有 `required_strategy_elements` 但文本无关键词），检测准确率 >= 85%

#### R-C-6: 主题门控（自动检测是否启用策略阶梯）（P1）

**功能描述**：
- 策略复杂度阶梯并非所有题材都需要（言情、日常向不需要 36 计）
- `NovelDirector.analyze_input()` 增加主题检测逻辑：
  - 输入：`genre`, `theme`
  - 输出：`enable_strategy_ladder: bool`
  - 检测规则（关键词匹配 或 LLM 判定）：
    - 若 `theme` 包含"战争""兵法""谋略""统一""争霸""帝王""策略""36 计"等关键词，启用
    - 若 `genre` 为"权谋""战争""历史""玄幻"（战争向）等，启用
    - 其他题材默认禁用
- 禁用时，`required_strategy_elements` 为空，Tier 映射跳过

**WHY**：
- 避免策略阶梯污染非战争题材小说（言情小说不需要"声东击西"）
- 自动检测减少用户配置负担

**验收标准**：
- 对战争题材小说（如本次 `novel_12e1c974`），`enable_strategy_ladder` = True
- 对言情题材小说（主题"都市霸总爱上我"），`enable_strategy_ladder` = False
- 检测准确率 >= 90%（10 个不同题材样本）

#### R-C-7: 策略使用统计与可视化（P2）

**功能描述**：
- 每章生成后，记录实际使用的策略元素（基于关键词匹配）
- 生成项目级统计报告：
  - 各 Tier 使用频次分布（柱状图数据）
  - 36 计使用覆盖率（已使用 X/36）
  - 策略重复率（同一元素在 N 章内重复使用次数）
  - 策略升级曲线（每 10 章的平均 Tier）
- 报告存储在 `workspace/novels/{novel_id}/strategy_report.json`，供 Web UI 展示

**WHY**：
- 可视化帮助作者/用户验证"策略是否真的在升级"
- 覆盖率统计验证元素库是否被充分利用

**验收标准**：
- 报告包含 Tier 分布、36 计覆盖率、平均 Tier 曲线
- 对生成 50 章的小说，36 计覆盖率 >= 60%（至少用了 21 计）
- 策略重复率 < 15%（同一元素在 5 章窗口内重复次数 / 总元素使用次数）

### 3.4 需求组 D：交叉集成与降级策略

#### R-D-1: 三机制协同触发（P0）

**功能描述**：
- 在卷高潮章节（`volume.climax` 对应的章节范围），同时触发三机制：
  - **卷进度**：高潮章节前 2-3 章，至少 1 个 `critical` 里程碑必须处于"待完成"状态（制造"必须在高潮解决"压力）
  - **系统失效**：高潮前 2-5 章安排系统失效（制造"无系统辅助下的高潮战"）
  - **策略升级**：高潮章节强制使用当前卷 `strategy_tier_range` 的最高 Tier 策略
- 示例：第 1 卷高潮（ch24-25 攻占青云山门）
  - ch22: 里程碑"攻占青云山门"进入倒计时（剩余 3 章）
  - ch23: 系统失效（`mode: degraded`，地图扫描范围缩小 70%）
  - ch24-25: 强制使用 Tier 3 策略（如"粮道封锁 + 内部分化 + 地形伏击"组合拳）
  - 效果：主角在系统受限情况下，必须依靠高级战术完成里程碑，张力最大化

**WHY**：
- 单机制作用有限，三机制叠加创造"完美风暴"
- 高潮是读者注意力峰值，应集中投放最强机制

**验收标准**：
- 对 6 卷小说的 6 个高潮章节，100% 满足"高潮前有系统失效 + 高潮章节有最高 Tier 策略"
- 人工阅读高潮章节，张力评分 >= 8/10（主观评价）

#### R-D-2: 无 LangGraph 时的降级执行（P0）

**功能描述**：
- 当前项目支持 LangGraph 缺失时的顺序执行 fallback（见 `src/novel/agents/graph.py`）
- 三机制必须在 fallback 模式下同样生效：
  - 卷进度追踪：不依赖 LangGraph，在每章生成后同步调用 `check_milestone_completion()`
  - 系统失效：不依赖 LangGraph，在 PlotPlanner/Writer 调用时读取 `system_failure_schedule`
  - 策略阶梯：不依赖 LangGraph，在 PlotPlanner 调用时读取 `strategy_tier_range`
- 所有数据存储在 `novel.json`（文件系统），不依赖 LangGraph 的状态管理

**WHY**：
- 项目设计哲学：LangGraph 是可选依赖
- 降级模式保证功能在所有环境可用

**验收标准**：
- 在 `langgraph` 未安装的环境下运行 `NovelPipeline.generate_chapters()`，三机制正常工作
- 卷进度 / 系统失效 / 策略阶梯的数据正确写入 `novel.json`

#### R-D-3: budget_mode 友好（省 Token）（P1）

**功能描述**：
- 三机制中涉及 LLM 调用的部分，优化 token 消耗：
  - **卷进度**：
    - 里程碑验证优先用关键词匹配（免费），仅复杂语义判断用 LLM
    - `continuity_brief` 的 `volume_progress` 提示块控制在 < 500 字
  - **系统失效**：
    - 失效排程在大纲阶段一次性生成（1 次 LLM 调用），后续章节只读取
    - 恢复条件优先用关键词检查，仅复杂条件用 LLM
  - **策略阶梯**：
    - 元素注册表预生成（1 次 LLM 调用），后续章节只随机抽取
    - 策略使用检查用关键词匹配（免费）
- 总 token 增量目标：平均每章 < 2000 tokens（相比当前流程）

**WHY**：
- 长篇小说 token 消耗巨大（200 章 × 平均 15k tokens/章 = 3M tokens）
- 省 token 策略是项目核心竞争力（见 CLAUDE.md）

**验收标准**：
- 生成 10 章样本，对比启用/未启用三机制的 token 消耗，增量 < 2500 tokens/章
- 里程碑验证中，关键词匹配覆盖率 >= 75%（只有 25% 需要 LLM）
- 策略元素抽取无 LLM 调用（纯随机 + 过滤）

#### R-D-4: 现有项目迁移路径（P1）

**功能描述**：
- 对已生成部分章节的项目（如 `novel_12e1c974` 已生成 27 章），支持平滑升级到新机制：
  - **卷进度**：
    - 运行 `migrate_add_milestones(novel_id)` 命令，基于现有 `volumes` 的 `theme/climax/end_hook` 生成里程碑
    - 对已生成章节，批量检查里程碑完成情况（LLM 回顾 + 关键词检查）
    - 标记已完成 / 未完成里程碑，未完成的里程碑计入下次生成的进度预算
  - **系统失效**：
    - 为剩余章节重新排程失效窗口（已生成章节不追溯）
    - 若用户希望追溯，可手动在 `failure_history` 中补录历史失效（可选）
  - **策略阶梯**：
    - 为剩余章节分配 Tier 范围
    - 对已生成章节做策略分析（关键词扫描），生成 `strategy_report` 作为基线
- 迁移命令：`python main.py novel migrate-narrative-arc workspace/novels/novel_12e1c974`

**WHY**：
- 不支持迁移 = 现有项目无法受益 = 用户流失
- 平滑迁移降低升级成本

**验收标准**：
- 对 `novel_12e1c974`（已生成 27 章）运行迁移命令，成功生成：
  - 6 个卷的里程碑（前 1 卷的已完成里程碑标记为 `completed`）
  - 剩余 173 章的系统失效排程
  - 剩余 173 章的策略 Tier 范围
- 迁移后继续生成 ch28-30，三机制正常工作

## 4. 非目标（明确排除）

- **不重写现有大纲模板**：`src/novel/templates/outline_templates.py` 保持不变，里程碑是在大纲生成后的增强层
- **不改变 LLM 提供者层**：`src/llm/` 保持不变，三机制仅消费 LLM 接口，不修改接口定义
- **不引入新的图数据库或向量库**：所有新数据存储在 `novel.json`（JSON 文件）或现有 SQLite `memory.db`
- **不支持用户自定义策略元素库**（MVP 阶段）：策略注册表由系统预生成，用户可读但不可编辑（V2 可扩展）
- **不支持多主角策略阶梯**（MVP 阶段）：策略阶梯绑定到小说整体，不区分不同主角的策略风格（单主角够用）

## 6. 成功指标

### 6.1 定量指标

- **卷进度健康度**：在 200 章小说中，各卷的里程碑完成率 >= 80%
- **系统失效执行率**：排程中的失效窗口，100% 在对应章节生效（无遗漏）
- **策略升级斜率**：前 50 章平均 Tier <= 2.5，后 50 章平均 Tier >= 4.5（战争题材）
- **策略覆盖率**：36 计在 100 章内覆盖率 >= 75%（27/36 计）
- **策略重复率**：同一元素在 5 章窗口内重复次数 / 总使用次数 < 15%

### 6.2 定性指标（人工审查）

- **卷切换流畅性**：卷边界前后 3 章，叙事尺度/主题/张力有明显转换（人工评分 >= 7/10）
- **系统失效张力**：系统失效章节的悬念/困境描写质量（人工评分 >= 7/10）
- **策略自然度**：策略使用是否自然融入情节，无生硬说教（人工评分 >= 7/10）

### 6.3 对比基线（改进幅度）

- 对比项目：`novel_12e1c974` 的 ch11-27（当前系统生成，无新机制）
- 改进目标：
  - 卷切换清晰度：从"模糊不清"→"明确可感知"
  - 系统依赖度：从"100% 依赖"→"失效章节 0% 依赖"
  - 策略多样性：从"4 种策略循环"→"20+ 种策略交替使用"

## 7. 风险与缓解

### 7.1 风险 1：里程碑过于刚性，扼杀创作灵活性

**缓解**：
- 里程碑设定为"目标范围"（如 ch15-20）而非"必须 ch18"
- `priority: normal` 的里程碑允许延后或放弃
- 用户可通过 API 手动调整里程碑

### 7.2 风险 2：系统失效过于频繁，破坏阅读体验

**缓解**：
- 失效密度可配置（默认每 20 章 1 次）
- 避让规则防止连续失效
- 失效模式多样化（60% `degraded`，只有 15% `offline`）

### 7.3 风险 3：策略元素库质量低，导致不合理策略

**缓解**：
- 元素库 LLM 生成 + 人工审核
- `usage_constraints` 过滤不合理策略
- 用户可反馈不合理元素，系统标记为"禁用"

### 7.4 风险 4：Token 消耗激增

**缓解**：
- 优先用关键词检查，LLM 仅用于复杂判断
- 里程碑 / 失效排程 / 元素库在大纲阶段一次性生成
- 每章增量 < 2500 tokens（已验证可行）
