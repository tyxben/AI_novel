# 架构审视报告 (2026-04-18)

> 由 reviewer agent 端到端审视输出。这是后续优化的诊断基础。
> 配套优化路线见 `ROADMAP.md`，模块利用率统计见 `MODULE_USAGE.md`。

## 项目目标（重新对齐）

> **提高作品的质量+流畅度，可以实现各种短/中长篇小说各种类型小说的写作。**

## 总评

> **方向只对了 30%。底层"记账"那一半（continuity_brief / 向量一致性 / changelog / impact_analyzer / reflexion）做得很认真，是真资产。但上层全部踩在三个错误的产品哲学上：**
>
> 1. 用工程指标替代审美判断（字数/AI 味/打分阈值都是硬编码常量）
> 2. Agent 数量替代 Agent 协作（9 个 Agent + 30+ Service，绝大多数在事后写报表，没有人在生成现场负责拍板）
> 3. 默认体裁是网文爽文（连 fallback 都写死"玄幻"）
>
> 结果就是测试 4671 个全绿、生成的小说一塌糊涂——因为测的都是字段格式，没人测"这章读着像不像人写的"。

## 用户脑中的"正确写作流程"（要对标的）

| 步 | 内容 |
|---|---|
| 1. 立项 | 一句灵感 → 体裁/主角/核心冲突/风格/长度 |
| 2. 搭骨架 | 世界观、主角配角、主线（3-5 句）、卷划分 |
| 3. 当前卷细纲 | 本卷章节列表 + 每章一两句 + 伏笔规划 |
| 4. 写一章循环 | 想清楚 → 出稿 → 重读 → 改 → 存 → 回顾教训 |
| 5. 推进卷 | 循环到本卷完，进下一卷 |
| 6. 中途回头改前章 | 判断范围 → 看涟漪 → 决定怎么改 → 执行+传播 → 挨个看后续是否被打脸 → 留痕 |

## 流程对标矩阵

| 用户 6 步 | 现状 | 评分 | 诊断 |
|---|---|---|---|
| 立项 | `novel write --genre --theme --target-words` | 🔴 | **反人类**——硬拆 3 字段，没有反问对话 |
| 搭骨架 | NovelDirector + WorldBuilder + CharacterDesigner | 🟠 | 能跑但不可逆——一次性烤大饼，中间没法打断改 |
| 当前卷细纲 | DynamicOutline (525 行) | 🟠 | 错位——是"全书 40 章一锅"，不是"先细化第一卷" |
| 写一章循环 | Writer + ContinuityService + Critic + Reflexion | 🟢 | 生成端最强环节，最近补的 critic+reflexion 是对的 |
| 推进卷 | 没有"卷"这个一等公民 | 🔴 | Volume 是空壳，没有"卷"的一等公民地位 |
| 中途改章 | smart-editor + impact_analyzer + 回滚 | 🟠 | 半成品——影响列出来了，但"挨个看后续是否被打脸"全靠手动 |

## 8 条最致命的"理念错位"

1. **把品味问题当成了规则问题**。AI 味黑名单、字数 2000-3000、节奏阈值、质量评分线、过度使用"仿佛/宛如"的检测——全是写死常量。这些恰恰是**作者的活**，不是工具的活。
2. **Agent 数量爆炸但没人负责"决定"**。9 个 Agent + 30 个 Service。生成时真正影响输出的只有 Writer 和 ContinuityService 两个，剩下的全是事后稽查或装饰。Director 在初始化后就退场了——没有"贯穿全书的导演"。
3. **观察层和控制层倒挂**。ObligationTracker / HealthService / KnowledgeGraph 都是在事后扫描章节，把统计结果写进 dashboard，**但下一章生成时不读它们**。"债务 87 条逾期"能被看见，却不能被预防。
4. **"章"是一等公民，"卷/弧线/场景"都是二等**。Volume 有 model 但 pipeline 不按 Volume 跑；Scene 有但每章固定 3 个 800 字；StoryUnit/Arc 有 model 没 service。
5. **测试覆盖维度错了**。4671 个测试主要在测"字段是否正确填充、JSON 是否能解析、回滚是否成功"。**没有一个测试在测"这一章读着像不像人话"**——这恰恰是产品的唯一目标。
6. **MCP 工具暴露的是"函数"不是"工作流"**。是数据库 CRUD 包装，不是创作工作流。AI 助手调它就是在替用户填表单，没有"我们一起想想这一卷该怎么收"的对话面。
7. **强 Schema 假设作者一开始就知道一切**。Outline / Character / World 都要求一次性填全。"第 8 章才决定加柳青鸾"是常态，但当前架构把它当 edge case。
8. **Reflexion + Critic 还在"自动重写"语义下**。quality_threshold 没达到就触发 Writer 重写——这就是替作者拍板。正确的语义是"提示作者：这章可能有问题，要不要看看"。

## 通用性陷阱（藏死的体裁/风格假设）

- **fallback genre 写死"玄幻"**（`novel_director.py` 多处），rhythm_templates 第一档就是"玄幻"。短篇悬疑/现实主义/恋爱日常 进来全部按爽文节奏跑。
- **三幕/四幕/起承转合被当成"模板可选"**，但 `cyclic_upgrade`（玄幻循环升级）才是 default。30 章悬疑硬塞进去就是灾难。
- **每章 3 scene × 800 字** 是网文写法。短篇可能一章一镜到底，文学小说可能一场戏写 5000 字。这个假设藏在 `writer.py` 里，外面不可调。
- **chapter.full_text + scenes 拼接** 假设所有章节都是 scene 化的。散文体、书信体、第一人称内心独白章节根本不该按 scene 拆。
- **AI 味黑名单不分体裁**：武侠里"目光如炬"是常用词，被一刀切；都市言情里"嘴角微微上扬"是合适的，也被一刀切。
- **target_words 在 Novel 级别，不在 Chapter / Volume 级别**。没法表达"高潮章 5000 字、过渡章 1500 字"。
- **质量评分模板是统一的 6 维**（quality_reviewer），不区分类型。悬疑要看"伏笔密度"，言情要看"情感曲线"，硬塞同一把尺。

## 一句话收尾

> **当前架构最该做的不是再加一个 Agent，而是砍掉一半 Agent 和所有硬阈值，把腾出来的空间还给作者，让工具退回到"放大镜 + 账本 + 跑腿"的本分。方向只对了 30%，但底层账本已经搭好了，掉头不算贵。**
